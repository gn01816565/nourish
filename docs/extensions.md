# 啾啾日常 — 延伸功能路線圖

> 版本：v1.0（2026-04-28）
> 文件擁有者：企劃 agent
> 適用階段：v0.1 MVP 完成後（19/19 ✅），規劃 v0.2 / v0.3 / App
>
> **前置條件**：MVP 已完成 4 大狀態、7 種終態、隨機事件（5 種）、成就（17 條）、圖鑑、開新蛋、互動動畫、進化粒子、成就解鎖粒子。
> 本文件目的：**避免想到什麼做什麼**，依 ROI / 工程量 / 留存影響排序，給主 agent 一份可直接挑工作的菜單。

---

## 評分維度說明

每個項目用以下五維量化（1–5 分，5 為最高/最大）：

| 維度 | 1 分 | 5 分 |
|------|------|------|
| **D1 留存** | 對首日留存無感 | 顯著拉高首日回訪 |
| **D7 留存** | 一週後不影響 | 一週後讓玩家還想開分頁 |
| **D30 留存** | 對長期黏著無效 | 一個月後仍是回訪主因 |
| **工程量** | <2 小時 | >40 小時 |
| **美術需求** | 零美術或可純 CSS | 需新角色 / 新場景 PNG |
| **風險** | 無副作用 | 可能破壞核心情感體驗 |

> **ROI 評估**：高 = D7+D30 ≥ 7 且 工程量 ≤ 3；中 = 兩者其一；低 = D 分數 < 5。

## 1. 即刻可做（v0.2，1–2 週工程量）

> 目標：**讓現有玩家黏更久 + 拓展可分享性**。每項都是純 JS/CSS/SVG，無新依賴。

### 1.1 季節性背景 + 限時食物（Seasonal Theme） 🎯⭐

- **描述**：依玩家機器本地日期切換背景（春櫻、夏海、秋楓、冬雪 4 套）+ 限時食物（春莓、夏冰、秋栗、冬糖）。每月 1 號自動切換，玩家收到 toast「迎接新季節！」+ 季節限定食物開賣 7 天。
- **核心情感**：「啾啾跟我一起過四季」→ 解決 GDD §12.2 假設「玩家覺得沒有結局失去動力」。
- **工程量**：6–8 小時。`bg-spring/summer/autumn/winter.svg` 4 張 + `CFG.seasonal` 表 + `getCurrentSeason()` 由 `Date.getMonth()` 推。
- **美術需求**：4 張背景 SVG（沿用現有 §1.3 色票即可，character-sheet 不衝突）。
- **ROI**：⭐⭐⭐⭐⭐ 高
- **D1/D7/D30**：3 / 4 / 5（D30 強，因每月會回來看新背景）
- **風險**：1 — 純疊加內容，不破壞核心。

### 1.2 老年寵物階段（Senior Stage）⭐

- **描述**：成雞達成終態後，再過 7 天進入「老年」階段。外觀微變（戴老花、毛色淡），新增 3 種獨享互動（曬太陽、回憶錄、餵蜂蜜水）+ 1 條「告別」終態圖鑑（玩家主動選「讓啾啾安詳退休」→ 自動歸檔到圖鑑 + 解鎖「相伴一生」成就 + 50 FC）。GDD §1.4 已暗示，這是該補的一塊。
- **核心情感**：陪伴感的最終昇華。**不是死亡**，是榮譽退休。
- **工程量**：8–10 小時。schema 加 `pet.seniorStartedAt`、3 個新互動、1 個新成就。
- **美術需求**：每種終態都要老年版 → **可暫時用「加眼鏡濾鏡」CSS filter** 蓋過，免新 SVG。等 PNG 換代再補。
- **ROI**：⭐⭐⭐⭐ 高
- **D1/D7/D30**：1 / 3 / 5（**D30 殺手**，給老玩家終局內容）
- **風險**：2 — 「告別」設計不慎易引發負面情緒，文案要溫暖（已研究過：Tamagotchi「化天使」是反例）。

### 1.3 截圖分享（Share Card） 📸

- **描述**：圖鑑頁加「📸 分享」按鈕，用 Canvas 把當前/歷代啾啾畫到一張 720×1280 卡片（含名字、終態、養成天數、用了幾次餵食、QR code 連回遊戲），玩家 long-press 存圖。卡片風格：暖黃底 + 啾啾居中 + 「我養出了【神雞】啾啾，6 天 ✨」slogan。
- **核心情感**：炫耀感（GDD §1.3 第 4 條），市場研究 §2 列為共通成功要素 ★★★★。
- **工程量**：6–8 小時。Canvas 2D 繪圖 + `toBlob` + 下載連結。
- **美術需求**：1 張卡片邊框模板 SVG（沿用色票）。
- **ROI**：⭐⭐⭐⭐⭐ 高
- **D1/D7/D30**：2 / 4 / 4（每次新終態都會想分享一次）
- **風險**：1 — 純輸出，無破壞性。
- **新玩家獲取**：分享出去的圖會帶 URL → 病毒傳播第一個樁。

### 1.4 每週挑戰（Weekly Quest） 📅

- **描述**：每週一 00:00 生成 1 個 7 天挑戰（從池中抽：「養 1 隻體力 >70 7 天連續」「7 天內餵食 30 次」「達成 2 種以上終態」），完成獎 200 FC + 限定裝飾（v0.3 配合裝飾系統）。每日任務不變，週挑戰是 D7 留存的硬鉤。
- **核心情感**：給「明確下一個短期目標」（已對成就講過，這是更密的版本）。
- **工程量**：5–7 小時。`state.weekly` schema + 5–10 個挑戰模板池 + 進度追蹤 hook。
- **美術需求**：0（純 UI）。
- **ROI**：⭐⭐⭐⭐ 高
- **D1/D7/D30**：2 / 5 / 3（D7 殺手，週挑戰天生對 D7 最有效）
- **風險**：1 — 純疊加。

### 1.5 多寵物同時養（Multi-Pet Slot） 🐤🐤

- **描述**：v0.2 開放「養第二隻」。需要花 200 FC 買「第二個雞舍」，可同時養 2 隻獨立進度的啾啾。切換用畫面右上「<」「>」翻頁。互動仍只能對「當前選中那隻」操作，不會同時點兩隻。
- **核心情感**：減少「我把第一隻收去圖鑑了會很捨不得」的心理摩擦。
- **工程量**：12–16 小時。`state.pets[]` 改陣列 + `state.activePetId`，所有 `state.pet.*` 改成 `getActivePet()`。schema bump v2。
- **美術需求**：1 張「雞舍 2」背景變體（小改即可）。
- **ROI**：⭐⭐⭐ 中
- **D1/D7/D30**：1 / 3 / 5
- **風險**：3 — schema 大改，需要嚴密 migration 測試。建議跟 1.2 老年階段一起 ship（同一波 schema migration）。

### 1.6 對話深度擴充（Dialogue Tree） 💬

- **描述**：當前「對話」只是 +2 心情 + emoji 氣泡。擴充為：根據「當前狀態 / 當前階段 / 已解鎖成就 / 連續登入天數 / 上次互動類型」從約 80 條對白池抽 1 條。例：早上開分頁 → 「啾啾打了個哈欠 🥱」、進化前 1 小時 → 「我覺得身體在發光…」、達成神雞後 → 「謝謝你照顧我，主人」。
- **核心情感**：讓啾啾「像活的」（GDD §12.2 待驗證假設）。市場研究 §2「情感擬人化」★★★★★。
- **工程量**：4–6 小時。對白池 JSON + `pickContextualLine(state)` 函式。
- **美術需求**：0。
- **ROI**：⭐⭐⭐⭐⭐ 高
- **D1/D7/D30**：4 / 4 / 3（D1 強，第一印象瞬間提升）
- **風險**：1 — 純內容擴充。

### 1.7 設定頁完工（Sound + I18n + Reduced Motion） 🔊

- **描述**：GDD §10.1 的最後一塊。補上：（1）音效開關 + 5 個 SFX（餵食啄聲、進化光波、洗澡水聲、心情氣泡 pop、按鈕 tap）；（2）i18n 骨架 zh-TW / en / ja 對白 JSON；（3）顯式 reduced motion toggle（已有 media query，但要可手動覆蓋）。
- **核心情感**：可訪問性 + 國際化舖路（Adopt Me 的全球化模式參考）。
- **工程量**：6–8 小時。WebAudio API（5 個短音檔即可，可用 jsfxr 線上產生 < 5KB 各檔）+ `t(key)` 函式 + 設定頁 UI。
- **美術需求**：0（音效需聲音資產）。
- **ROI**：⭐⭐⭐ 中
- **D1/D7/D30**：3 / 2 / 2（不是 D 殺手，但是「上線前必補」的 polish）
- **風險**：1 — 必補項目。

### 1.8 進化前兆（Pre-evolution Glow） ✨

- **描述**：進化倒數 < 1 小時時，啾啾身體周圍開始有微金光脈動（CSS `box-shadow` 動畫即可），speech 池增「我覺得身體在發光…」，並在 footer 顯示「✨ 即將進化…」。讓玩家產生「想守在線上看進化」的衝動。
- **核心情感**：儀式感前奏（市場研究 §2「儀式感事件」★★★★★）。
- **工程量**：2–3 小時。
- **美術需求**：0。
- **ROI**：⭐⭐⭐⭐ 高（CP 值極高）
- **D1/D7/D30**：3 / 4 / 2（進化在第 5–7 天，剛好是 D7 留存生死線）
- **風險**：1。

### v0.2 總工程量估算

全部做完 ≈ 50–60 小時（約 7–9 個 work day）。

**建議 ship 順序**（依 ROI/工程量比）：

1. **第 1 週 ship**：1.6 對話深度 + 1.8 進化前兆 + 1.3 截圖分享（合計 ~16h，全是高 ROI 低工程量）
2. **第 2 週 ship**：1.1 季節背景 + 1.4 每週挑戰 + 1.7 設定頁完工（合計 ~22h）
3. **第 3 週起**：1.2 老年階段 + 1.5 多寵物（合計 ~28h，schema 改動較大）

## 2. 中期路線（v0.3，1 個月）

> 目標：**從個人遊戲擴張到輕度社群 + 內容深度**。引入第一波變現實驗。

### 2.1 裝飾物商店（Decoration Shop）🎩

- **描述**：用 FC 買帽子、圍巾、背景物件（狗骨頭、書架、麥克風架），純 SVG 疊圖，不影響數值。每個裝飾 50–500 FC。3 套：基礎（10 件）、季節限定（每季 5 件）、成就解鎖款（達成 streak_30 解鎖金牌徽章等，不可購買）。
- **GDD 對應**：§6.1 已預留消耗用途。
- **工程量**：20–30 小時。雞舍佈局系統（拖拉放置）+ 商店 UI + 預覽功能。
- **美術需求**：~25 個 SVG 裝飾品（沿用色票，不需新角色）。
- **ROI**：⭐⭐⭐⭐ 高
- **D1/D7/D30**：2 / 4 / 5（裝飾收集 = D30 王者）
- **風險**：2 — 需設計好「FC 通縮」防止後期玩家錢花不完。
- **變現預備**：v0.3 末可選 1–2 個「神雞限定皇冠」開放 IAP 測水溫（紙幣 NT$30 等值），但**不影響數值**。

### 2.2 推薦碼 + 邀請朋友（Referral） 🎁

- **描述**：每個玩家有 6 字推薦碼。新玩家輸入碼 → 雙方各得 100 FC + 限定徽章「介紹人」。Web 版用 URL 參數（`?ref=ABC123`）+ localStorage 記錄。**沒有後端**，純客戶端互信，真正防作弊等 v1 雲端版。
- **核心情感**：給核心玩家炫耀的硬鉤 + 病毒傳播的法律機制。
- **工程量**：8–10 小時。
- **美術需求**：1 張徽章 SVG。
- **ROI**：⭐⭐⭐⭐ 高
- **D1/D7/D30**：5 / 3 / 2（D1 殺手，新玩家獲取）
- **風險**：3 — 純客戶端可被刷，但 MVP 階段不擔心。

### 2.3 雲端存檔（OAuth + Save Sync）☁️

- **描述**：Google / Apple Sign-in，存檔同步到 Firebase（免費額度足夠 < 10K MAU）。解決市場研究反模式表「無存檔遺失保護」。
- **核心情感**：**降低留存風險**（清快取就消失是當前最大流失原因之一）。
- **工程量**：30–40 小時。Firebase 整合 + login flow + conflict 解決（兩裝置同時養，merge 策略）。
- **美術需求**：1 個登入 modal。
- **ROI**：⭐⭐⭐ 中（防流失，不是拉新）
- **D1/D7/D30**：1 / 2 / 5（D30 防斷崖）
- **風險**：3 — 第一次引入後端依賴，違反「即玩即走」設計理念，需謹慎，建議用「可選功能」設計（不登入也能玩）。

### 2.4 社群圖鑑 / 排行榜（Public Dex） 🌐

- **描述**：搭配 2.3 雲端存檔，公開個人圖鑑頁（URL `/p/<playerId>`），可看朋友養出哪些終態。可選擇「公開 / 朋友 / 私人」。輕度排行榜：神雞達成最快、連續登入最久、收集 7 種最快。**不放戰力或氪金榜**（避免變一般 mobile game）。
- **核心情感**：被看見 + 軟性炫耀（市場研究 Adopt Me 案例 §1.5）。
- **工程量**：15–20 小時。
- **美術需求**：1 套排行榜 UI。
- **ROI**：⭐⭐⭐ 中
- **D1/D7/D30**：2 / 3 / 4
- **風險**：3 — 引入社交壓力，需控制好「對比焦慮」。建議只放「達成」榜，不放「未達成」排序。

### 2.5 食物料理系統（Cooking Mini-game） 🍳

- **描述**：成雞階段解鎖小廚房，玩家可用 2–3 個基礎食材（玉米 + 莓果）合成「莓果玉米派」（飢餓 +50 + 心情 +20），合成需小遊戲（節奏點擊 3 次成功 = 完美料理 + 加成；失敗 = 普通效果）。
- **核心情感**：增加互動深度，避免「點按鈕無聊」。
- **工程量**：15–20 小時。料理譜表 + 節奏小遊戲 + 食材庫存 schema。
- **美術需求**：~10 張料理成品 SVG。
- **ROI**：⭐⭐⭐ 中
- **D1/D7/D30**：1 / 4 / 4
- **風險**：2 — 增加遊戲複雜度，需評估是否衝擊「5 分鐘 micro loop」定位。

## 3. 長期願景（App 階段，半年後）

> 目標：**從網頁延伸到原生體驗 + 深度商業化**。需要 backend / 平台店面 / 較大資源。

### 3.1 PWA + 推播（Push Notifications） 🔔

- **描述**：包成 PWA（manifest.json + service worker），可安裝到 home screen。當小雞飢餓 < 20 持續 30 分鐘 → 系統推播「啾啾餓了～」。**重點**：玩家可控制推播頻率 / 全關（避免變成騷擾）。
- **核心情感**：市場研究 §5「推播是最強回訪驅動」。
- **工程量**：20–30 小時（含 service worker、推播伺服器最小實作）。
- **美術需求**：app icon 多尺寸、splash screen。
- **ROI**：⭐⭐⭐⭐⭐ 高
- **D1/D7/D30**：3 / 5 / 5（推播 = 留存核武器）
- **風險**：3 — 需後端推播服務（Firebase Cloud Messaging 免費額度足夠）。

### 3.2 IAP 純裝飾（純美觀，不影響數值）💎

- **描述**：v0.3 已試水的延伸。3 個價格帶：NT$30（單裝飾）、NT$120（季節包）、NT$330（全裝飾月卡）。**禁忌**：不能加速進化、不能多餵食、不能跳冷卻。所有付費玩家拿到的優勢都是視覺性的。
- **核心情感**：保留市場研究反模式「逼氪卡進度」的迴避策略。
- **工程量**：30–50 小時（含金流、收據驗證、退款）。
- **美術需求**：高，需 30+ 件高品質裝飾，**這時候 character-sheet § 6 的 PNG 升級必須完成**。
- **ROI**：⭐⭐⭐ 中（變現，非留存）
- **D1/D7/D30**：- / - / -（變現指標：ARPU、付費率，非留存）
- **風險**：4 — 平台抽成 30% + 退款處理 + 兒童付費保護。

### 3.3 AR 模式（把啾啾放桌上）📱

- **描述**：iOS QuickLook + Android ARCore，把 3D 化的啾啾放在玩家桌面，可拍照上傳。Niantic Peridot 已驗證可行。
- **核心情感**：把寵物從螢幕「帶到現實」。
- **工程量**：60–100 小時（含 3D 建模、AR 整合）。
- **美術需求**：**極大**，需從 2D SVG 升級到 3D 模型（USDZ + glb 兩格式）。
- **ROI**：⭐⭐ 低 ROI 但⭐⭐⭐⭐⭐ 行銷亮點
- **D1/D7/D30**：2 / 2 / 2（D 不漂亮，但可發 PR）
- **風險**：4 — 跨平台 AR 維護成本高，3D 又會破壞 2D 風格。建議**只做拍照模式，不做即時操作**。

### 3.4 AI 對話（接 LLM）🤖

- **描述**：玩家可打字跟啾啾聊天，啾啾用 GPT-4o-mini / Claude Haiku 等便宜 LLM 回應，**人格鎖死**（系統 prompt 寫死「你是隻黃色小雞，個性害羞期待、用 30 字內中文短句」）。
- **成本估算**：Haiku 每千次對話 ~NT$3，月活 1 萬人 × 月均 20 次 = 20 萬次/月 = NT$600。**可承受**，但要設 daily quota（每人每天 5 次）防濫用。
- **核心情感**：**真正讓啾啾「活著」**。Talking Tom 5 億下載驗證需求。
- **工程量**：40–60 小時（含 backend proxy、quota、moderation API）。
- **美術需求**：聊天 UI。
- **ROI**：⭐⭐⭐⭐ 高（情感深度）
- **D1/D7/D30**：4 / 4 / 4
- **風險**：5 — LLM 可能回不在角色內的話 / 跨入敏感話題。**必須**用 moderation API + 預設拒絕模板。

### 3.5 跨遊戲匯入（Adopt Me / Tamagotchi 致敬模式）🎮

- **描述**：玩家輸入 Tamagotchi 序號 / Adopt Me 寵物 ID，啾啾根據對應寵物資料生成「致敬款」外觀（不違反版權，是粉絲改編風格）。
- **核心情感**：勾住懷舊客群 + 跨 IP 玩家。
- **工程量**：20–30 小時（無官方 API，純玩家自填）。
- **美術需求**：~20 個致敬款變體。
- **ROI**：⭐⭐ 低（小眾）
- **D1/D7/D30**：2 / 2 / 1
- **風險**：4 — IP 風險，需法務 review。**建議僅做「靈感款」不做「複刻款」**。

## 4. 不該做（避坑清單）

> 這些功能乍看誘人，但會破壞核心體驗或引入難以承擔的風險。**主 agent 收到這類提案請拒絕**。

### 4.1 ❌ 真實死亡 / 強制送葬

- **理由**：MVP 設計哲學第一條（GDD §1.3 + CLAUDE.md §10）。
- **失敗案例**：早期 Tamagotchi 死亡是「儀式感」，但現代輕度玩家市場已驗證**會直接棄坑**。市場研究 §3 反模式表第 1 條。
- **就算用「自然老去」溫柔包裝也 NO**。1.2 「老年階段」是上限。

### 4.2 ❌ PvP 戰鬥 / 排名榜

- **理由**：違反 GDD §1.2 目標玩家「不想要 PvP 壓力」+ §12.3「不焦慮」原則。
- **失敗案例**：Cookie Run / 慕戰 等養成 PvP 化都會驅離輕度療癒玩家。
- **就算是「友善 PK 友軍」也 NO**。2.4 「成就排行榜」是極限（且只放正向達成）。

### 4.3 ❌ 開蛋轉蛋付費（Gacha）

- **理由**：違反 GDD §10 「禁止強制催氪」。
- **失敗案例**：Adopt Me 引入轉蛋後，輕度玩家流失到 Roblox 其他遊戲。
- **3.2 IAP 限定純裝飾**就是極限。

### 4.4 ❌ NFT / 區塊鏈 / 寵物交易市場

- **理由**：市場研究 §5「不建議做」明確列出。
- **失敗案例**：所有引入 NFT 的養成遊戲（Axie Infinity）都把「養成情感」異化為「投機資產」。
- **這條在「養成情感投射」遊戲是核武級反模式**。

### 4.5 ❌ 多人即時互動（你家啾啾跑到我家玩）

- **理由**：需要實時 WebSocket + 同步狀態 + 防作弊，工程量爆炸（~200 小時），且不在目標玩家的「分頁旁掛著一個人」情境內。
- **替代方案**：2.4 公開圖鑑頁（非同步社交）已能滿足看朋友養什麼的需求。

### 4.6 ❌ 真人語音陪伴（Talking Tom 升級版）

- **理由**：麥克風權限 + 隱私法規（GDPR / CCPA）成本極高。**18–35 歲核心客群會直接拒絕授權**。
- **替代方案**：3.4 LLM 文字對話已能達成「擬人化」效果。

### 4.7 ❌ 強制每日上線打卡 / 連續斷掉重置

- **理由**：違反 GDD §12.3 「不焦慮」原則。連續登入要「鼓勵」，不要「處罰」。
- **錯誤示範**：「斷簽後從第 1 天重新計算」會驅離出差/旅行玩家。
- **建議改**：斷簽後保留歷史最高紀錄為徽章，新一輪重新計但不影響獎勵級數。

### 4.8 ❌ 同步引入太多新終態（一次加 5 個分支）

- **理由**：character-sheet § 8 的 5 道 gating 一次只能消化少量產出。一次加 5 個會讓「畫風一致性」失守。
- **節奏**：v0.2 不加新終態（先做完現有 7 種的 PNG 升級），v0.3 加 1–2 個（如「龐克雞」「廚師雞」）。

## 5. 推薦下個 sprint 重點 3 件事（給主 agent）

> 假設 1 sprint = 1 週（~25 工時）。**已按工程量打包成可一次 ship 的組合**。

### 🥇 第一名：**對話深度 + 進化前兆 + 截圖分享** （合計 ~16h）

- **為什麼是這 3 個一起**：
  - 全部低工程量、全部高 ROI、全部不需 schema migration
  - 三者協同效應強：對話讓啾啾「像活的」→ 玩家更想看進化 → 進化前兆讓玩家守在線上 → 看到神奇終態 → 立刻想分享截圖
  - **D1/D7 雙殺**：對話是 D1 第一印象、進化前兆是 D7 鉤子、截圖是病毒擴散
- **具體任務**：
  1. 寫 80 條 contextual dialogue 池（按 6 種狀態 × 3 階段 × 4 種事件分類），加 `pickContextualLine(state)`
  2. CSS keyframes 加 `pre-evolve-glow` + speech 池新增 3 條進化前兆台詞 + footer 倒數提示
  3. Canvas 2D 畫卡片（720×1280）+ 圖鑑頁加分享按鈕 + URL 帶 `?ref=` 參數
- **驗證**：開瀏覽器手動測 3 個場景：剛開新蛋（看對話多樣性）、進化前 1 小時（看金光）、達成終態後（按分享生成卡片）
- **預期效果**：D7 留存 +15%、分享率從 0 → 5%

### 🥈 第二名：**季節背景 + 每週挑戰 + 設定頁完工** （合計 ~22h）

- **為什麼是這 3 個一起**：
  - 設定頁是 GDD §10.1 最後一塊（補完 v0.1 才能算正式）
  - 季節 + 週挑戰都是「規律性回訪鉤」，跟設定頁的「音效」搭配可顯著提升 polish 感
  - **D7/D30 都被照顧到**：週挑戰殺 D7、季節殺 D30
- **具體任務**：
  1. 補 4 張季節背景 SVG + `getCurrentSeason()` + `CFG.seasonal` 表
  2. 5–10 個週挑戰模板池 + `state.weekly` schema + 進度 hook
  3. 5 個 SFX（jsfxr 線上產生）+ 音效開關 + i18n 骨架（zh-TW / en，先不做 ja）
- **驗證**：本地調系統時間到下個月 1 號 → 看背景切換；連續 7 天養成 → 看週挑戰刷新
- **預期效果**：D7 +10%、D30 +20%、玩家好感度 +1 個量級（音效 polish 是體感分水嶺）

### 🥉 第三名：**老年階段 + 多寵物欄位**（合計 ~28h，schema bump v2）

- **為什麼一起做**：
  - 都需要 schema migration → 一次 bump 比兩次划算
  - 兩者**互補**：老年階段給「終局內容」、多寵物給「不捨得歸檔」的解法
  - 此時 v0.2 內容已經充實，玩家有理由「多養幾隻同時體驗不同分支」
- **具體任務**：
  1. schema v1 → v2 migration（`pet` → `pets[]`、加 `seniorStartedAt`）
  2. 「養第二隻」雞舍購買 + 切換 UI（< / > 翻頁）
  3. 老年階段 3 個獨享互動（曬太陽 / 回憶錄 / 餵蜂蜜水）+「告別」流程
  4. 1 個新成就：相伴一生（讓啾啾安詳退休）
- **驗證**：用 debug 模式快速跳到成雞 → 等 7 天（或 debug 跳階段）→ 觸發老年階段；同時測 schema migration 對舊存檔安全性
- **預期效果**：D30 +25%（老玩家終局內容）、玩家流失點從「沒事做」延後 2–3 週
